# Migration: v0.6.1 → v0.6.2

## Summary

Muscle activation fields consolidated. Three redundant frontmatter fields (`triggers`, `keywords`, `topic`) merged into a single `triggers` list at parse time. Old format still works — the parser reads all three and merges them — but the canonical format is now just `triggers`.

**Breaking:** No. Old muscles with `keywords:` and `topic:` will continue to work. The parser merges them into `triggers` automatically. This migration is about cleanup, not survival.

## Changes

### 1. Muscle frontmatter: triggers consolidation

**Before (v0.6.1):** Four fields for activation matching with different score weights.
```yaml
triggers: [dev, ship, release]      # score: 10
keywords: [ship, commit, push]      # score: 5
tags: [dev, workflow]               # score: 5
topic: [dev, workflow]              # score: 4
```

**After (v0.6.2):** Single `triggers` field. `tags` stays for categorization only.
```yaml
triggers: [dev, ship, release, commit, push, workflow]
tags: [dev, workflow]
```

**Why:** The fields were redundant — "focus" in `triggers`, `keywords`, `tags`, AND `topic` for the same muscle. One list, one purpose.

### 2. New frontmatter field: `tools`

Muscles can now declare which scripts they reference:
```yaml
tools: [soma-ship.sh, soma-verify.sh]
```

When the agent runs one of these scripts, the muscle's heat bumps automatically. No configuration needed — the existing `tools:` field many muscles already had is now parsed and used.

### 3. Natural heat detection

Muscles now get heat bumped from:
- **Script execution** — running a script listed in the muscle's `tools:` field
- **File path matching** — editing a `.svg` file bumps muscles with `"svg"` in triggers
- **Path segment matching** — editing `content/blog/foo.md` bumps muscles with `"blog"` in triggers

Previously, heat only bumped via `/pin`, `soma focus`, or reading the muscle file directly.

### 4. settings.json: version field

New `version` field in settings.json tracks which Soma version the workspace was last migrated to:
```json
{
  "version": "0.6.2",
  ...
}
```

## Migration Steps

### Step 1: Add version to settings.json

If `settings.json` has no `version` field, add it:

```json
{
  "version": "0.6.2",
  ...
}
```

**Detection:** `!settings.version`
**Action:** Set `settings.version = "0.6.2"`

### Step 2: Consolidate muscle frontmatter (optional cleanup)

For each muscle in `amps/muscles/`:

1. Read `triggers:`, `keywords:`, `topic:` from frontmatter
2. Merge all values into `triggers:` (deduplicate, lowercase)
3. Remove `keywords:` and `topic:` lines
4. Keep `tags:` unchanged (categorization, not activation)

**Detection:** Muscle has `keywords:` or `topic:` in frontmatter
**Action:** Merge into `triggers:`, remove old fields
**Risk:** Low — parser already handles both formats. This is cleanup only.

### Step 3: Add tools field to muscles (optional enhancement)

For muscles that reference scripts in their body but don't have `tools:` in frontmatter, add it:

```yaml
tools: [soma-ship.sh, soma-verify.sh]
```

**Detection:** Muscle body contains `soma-*.sh` references but no `tools:` frontmatter
**Action:** Extract script references, add `tools:` field
**Risk:** None — additive only

## Verify

After migration:

1. `npm run check` — zero type errors
2. `npm test` — all suites pass
3. `soma focus <keyword>` — muscles still match correctly
4. Run any script → check that related muscle heat bumped (enable debug: `soma debug on`)

## Notes

- The parser is backwards-compatible: old format works indefinitely
- `keywords:` and `topic:` in frontmatter are still parsed if present — they get merged into `triggers`
- Hub content (community repo) may still have `keywords` and `topic` in hub-index.json — these are read-only metadata for search, not activation
- This migration map was the first one created — it establishes the pattern for future versions
