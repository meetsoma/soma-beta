# Migration: v0.6.2 → v0.6.3

## Summary

Hub operations consolidated into new `/hub` command (soma-hub.ts extension). The `/install` and `/list` commands are now aliases that redirect to `/hub`. Scripts can be installed from the community hub. git-identity protocol archived from bundled set.

**Breaking:** No. `/install` still works as a backward-compat alias that notifies users to use `/hub install` instead. All existing `.soma/` content continues to work unchanged.

## Changes

### 1. New extension: soma-hub.ts

All community hub operations consolidated under `/hub`:

```
/hub install <type> <name> [-g|-p] [--force]  — install from community
/hub fork <type> <name>                       — fork with lineage tracking
/hub share <type> <name>                      — share to community (needs gh)
/hub find <keywords>                          — search hub content
/hub list [type] [--remote]                   — browse content
/hub status                                   — hub connection info
```

**Install flags:**
- `-g` — install globally to `~/.soma/` (default)
- `-p` — install to project `.soma/` (requires project-level .soma/)
- `--force` — overwrite existing content

**Security:** Hub operations are hardcoded in the extension (not drop-in scripts) to prevent tampering with install sources, fork lineage, or privacy scanning.

### 2. Script install support

Scripts can now be installed from the community hub:

```
/hub install script soma-code
```

This downloads `scripts/soma-code/soma-code.sh` from the community repo, saves to `.soma/amps/scripts/soma-code.sh`, and makes it executable (`chmod +x`).

### 3. `/install` → backward-compat alias

The old `/install` command still works but shows a migration notice:

```
💡 /install is now /hub install
   Running: /hub install protocol maps
```

Same for `/list` → `/hub list`.

### 4. Core protocol scope

Five protocols whose behavior is coded in TypeScript extensions are now `scope: core` instead of `scope: bundled`. Core protocols are documentation — they're readable on demand but don't load into the system prompt, saving ~2000 tokens. Affected: `breath-cycle`, `heat-tracking`, `session-checkpoints`, `git-identity`, `hub-sharing`.

`git-identity` was restored (previously archived) as `scope: core` — it documents the `guard.gitIdentity` setting which now supports email arrays for multi-account users.

### 5. Preload template override (optional)

Place `.soma/prompts/preload-template.md` to customize the exhale/breathe preload format. Template variables (`{{today}}`, `{{sessionId}}`, etc.) are interpolated at render time. No template file = current default behavior. This is opt-in — no migration action needed.

## Migration Steps

### Step 1: Ensure soma-hub.ts extension is available

**Detection:** `soma-hub.ts` not found in extension loading paths
**Action:** `npm install -g meetsoma@latest` or `soma-dev switch dev`
**Risk:** Without the extension, `/hub` commands won't register. `/install` alias will show a notice but can't delegate.

### Step 2: Update settings.json version

**Detection:** `settings.version !== "0.6.3"`
**Action:** Set `version: "0.6.3"` in settings.json
**Risk:** None — version field is informational for migration chaining.

### Step 3: Update core protocols to scope: core

**Detection:** `breath-cycle.md`, `heat-tracking.md`, `session-checkpoints.md`, `git-identity.md` have `scope: bundled`
**Action:** Change `scope: bundled` → `scope: core` in each. Restore `git-identity.md` from `_archive/` if previously archived.
**Risk:** None — core protocols were documentation of coded behavior. Removing them from prompt injection saves tokens without changing agent behavior.
