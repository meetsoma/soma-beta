# Migration: v0.23.0 → v0.23.1

## Summary

**Auto-breathe is now opt-in by default.** Existing users with
`settings.breathe.auto: true` are flipped to `false` once on next boot or
`soma doctor` run, then a sentinel is recorded so the migration never
re-runs. Users who want auto-breathe back can re-enable it manually —
the migration respects that choice.

This is also the migration that introduces the `settings.migrations[]`
tracker — a small array of applied one-time-fix IDs that lets future
semantic migrations flip existing values exactly once without needing
a per-version executable script.

## Why

`breathe.auto: true` causes a proactive context rotation at `rotateAt%`
(default 70%). The rotation tears down the current TUI and mounts a fresh
one. **If the user is mid-compose when it fires, their typed-but-unsent
message is wiped with no warning.** Reported live in s01-236eb4: Curtis
was composing a follow-up at ~70% and the rotation ate the message.

Manual `soma exhale` / `/breathe` does the same thing without surprise.
Auto-breathe is still useful for unattended long-running workflows, so
it stays as opt-in — just no longer the default for interactive use.

## Behavior

### For new installs (fresh `soma init`)

No change. `BUILTIN_SETTINGS` in `core/init.ts` doesn't include a
`breathe` block. The first boot's Tier 1 `addIfMissing` adds
`breathe: { auto: false, triggerAt: 50, rotateAt: 70, graceSeconds: 30 }`.
Users start with auto OFF, as before.

### For existing users with `breathe.auto: true`

On next boot OR next `soma doctor` invocation:

1. The Tier 1 migration block detects `current.breathe.auto === true`.
2. Flips it to `false`.
3. Adds `"breathe-auto-off-v0.23.1"` to `settings.migrations[]`.
4. `soma doctor` prints a one-line notice;
   soma-boot logs to debug only (no boot-time noise).
5. Subsequent boots skip the migration (sentinel present).

### For users who manually re-enable

If the user later sets `breathe.auto: true` again, the sentinel stays
in `migrations[]` and the migration does NOT run again. Re-enable is
respected.

## Actions

**soma will automatically apply this migration on next boot (`before_agent_start`).** No manual steps required.

1. At boot, `soma-boot.ts` checks `settings.migrations[]` for `breathe-auto-off-v0.23.1`.
2. If absent AND `breathe.auto === true` → flip to `false`, append sentinel.
3. Sentinel recorded → migration never re-runs.
4. User who re-enables `breathe.auto: true` after migration: respected forever.

## Migration Steps

**Mechanical — handled by Tier 1 / `soma doctor`. Nothing for the user
to do.** If you have a project with `breathe.auto: true`, it will flip
to `false` on next boot.

If you actively want auto-breathe back after the migration runs, edit
`.soma/settings.json`:

```json
{
  "breathe": {
    "auto": true
  }
}
```

The migration sentinel respects this — it won't be re-flipped.

## Implementation references

- `extensions/soma-boot.ts` — Tier 1 migration block, `applyOnce()`
  helper (lines ~1228-1258).
- `npm/thin-cli.js` — `soma doctor` mirror (lines ~1055-1080).
- Both sites share the same migration ID (`breathe-auto-off-v0.23.1`)
  so whichever runs first records the sentinel; the other becomes a
  no-op.
- Tracker: `settings.migrations: string[]` — append-only list of
  applied migration IDs.

## Future migrations

The `applyOnce(id, fn)` pattern is reusable. To add a one-time fix:

1. Append the migration block to **both** `soma-boot.ts` Tier 1 and
   `npm/thin-cli.js` doctor (keep them in sync — first to run wins).
2. Pick a stable ID like `<topic>-<verb>-v<X.Y.Z>`.
3. The function returns `true` if it actually changed something (for
   the `fixes++` counter), `false` if the precondition wasn't met
   (e.g. value was already correct).
4. Document it here in the version's migration map.
