# v0.11.0 → v0.12.0 — Somaverse Edition

> Backfilled s01-f6e928 (2026-04-22). This map consolidates every patch between
> 0.11.0 and 0.12.0 (0.11.1, 0.11.2, 0.11.3, 0.11.4, 0.12.0) because none of
> them required a user-side `.soma/` migration. See `CHANGELOG.md` for per-patch
> detail.

## Summary

The agent learned to pair with Somaverse. All `.soma/` changes are **additive
and opt-in** — no breaking changes. Existing projects keep working; the new
capabilities activate only when the user runs `soma login`.

## What Changed

### User-facing
- **`soma login`** — pairs your agent with Somaverse via a device key stored at `~/.soma/device-key`. Opt-in.
- **Hub-connect extension** — connects your agent to the Somaverse hub as a provider. Runs alongside `bridge-connect` (local) with dedup.
- **28 agent tools** via hub-connect — 10 workspace, 10 browser, 5 AI, 2 plugin state, 1 browser_links. Auto-route to hub when `~/.soma/device-key` exists, fall back to localhost.
- **Device-key auth** — 192-bit keys, Argon2-hashed server side. Per-user workspace isolation.

### Stability + cost (0.11.1 → 0.11.4)
- **Cache health tracking** — statusline shows `cacheRead / cacheWrite / cost` + invalidation counter. Alerts on >50K-token cache writes.
- **Idle session auto-shutdown** — post-exhale (15 min) + absolute (30 min). Independent of keepalive.
- **Image payload guard** — strips old images when >15 accumulate; pauses keepalive on 400/invalid_request.
- **Image budget** — soft notify at 8 images, hard auto-compact at 10. Configurable via `imageBudget.softAt` / `hardAt`.
- **Breathe token cap** — `breathe.maxTokens` caps effective context window for threshold math (fixes dormant breathe on 1M-ctx models).
- **Error handling** — OAuth vs API-key aware. Account rate limits (plan limit) separated from transient extra-usage errors. Billing messages sanitized at build time.
- **Pi runtime 0.64.0 → 0.67.1** — fixes long-session stack overflow (#2651), queued-message flush, subscription auth warning. Telemetry disabled (`PI_TELEMETRY=0`).
- **Removed 3 never-implemented settings** — `memory.flowUp`, `sessions.overwriteGuard`, `checkpoints.project.workingBranch`. Default `settings.version` bumped 0.6.4 → 0.11.3.

### Hygiene
- **Script description parser** — `getScriptMeta()` now skips YAML frontmatter; stale `---` descriptions in the system prompt are gone.
- **Build pipeline** — `build-dist.mjs` now reads Pi from `node_modules/` instead of stale `repos/cli/dist/`. Root cause of months of invisible dist drift.
- **Release gate** — `soma-release.sh` blocks on `dist/` vs `node_modules/` drift via `soma-dev verify upstream`.


**Actions:**
- No user-side action required — `addIfMissing` in `extensions/soma-boot.ts:1167-1178` adds new settings keys on next boot.
- Doctor tier-1 soft-check scaffolds missing files via `writeIfMissing` (never clobbers).
- See § Migration Steps below for optional opt-in features.

## Migration Steps

**None required for existing users.** Everything lands via `addIfMissing` at boot
(see `extensions/soma-boot.ts:1167-1178`). The three removed settings were
never read by any code path, so their presence in old `settings.json` files is
harmless — doctor does not rewrite them, they're simply ignored.

### Optional (users who want Somaverse)

1. `npm install -g meetsoma@0.12.x` (or `soma update`)
2. `soma login` — pair with somaverse.ai

### Optional (users who want cache observability)

No action — cache-health tracking activates automatically in the statusline.

## Compatibility

- `.soma/settings.json` — forward-compatible. `version` field auto-advances on next `soma update`.
- Custom extensions — no API changes for local use. Hub mode is opt-in via `~/.soma/device-key` presence.
- `SOMA.md` / `identity.md` continue to work as identity fallbacks (soul.md primary since 0.11.0).
