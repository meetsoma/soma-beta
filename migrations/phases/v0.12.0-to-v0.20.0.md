## Summary

Soma learned to delegate. A parent agent can now spawn focused child agents
with role-tuned prompts and restricted tool sets. The CLI was split into
three layers (thin-cli → personality → core) and Pi-native prompt
compilation replaced custom string manipulation.

All `.soma/` changes are **additive**. A project that upgrades without
touching its body stays functional — new features sit dormant until used.

## What Changed

### Delegation (the headline)
- **`delegate` tool** — spawns a child soma with a role from `body/children/*.md`. Inherits parent identity (soul/voice/ecosystem) but runs with role-specific system prompt.
- **7 child roles shipped** — `general`, `verifier`, `builder`, `curator`, `planner`, `doc_writer`, `reflector`. Seeded in `body/children/` at install time.
- **Curator loop** — verifier → builder → curator → apply. Parent dispatches tasks; curator reconciles output. `--auto-apply` for hands-off mode.
- **Child invocations log** — `state/invocations/<role>/` tracks every dispatch (prompt, output, cost, latency).
- **Scratchpad** — shared workspace under `state/scratch/` for multi-child coordination.

### Tool registry
- **`somaRegisterTool()`** — man-in-the-middle between extensions and Pi's registry. Preserves `promptSnippet` + `promptGuidelines` in the compiled system prompt (Pi's `ToolInfo` strips them).
- **`_tools.md` configuration** — new body-chain file. Sections: `## Disabled` (opt-out list), `## Overrides` (per-tool field tweaks), `## Custom` (parsed; registration lands 0.20.3). Hardwired set (`delegate`) cannot be disabled.
- **3 new Soma tools** — `context_status` (runway), `file_outline` (cheap orientation), `search` (ripgrep + Brave API).
- **`docs/tools.md`** (new) — tool authoring reference.

### Prompt compilation
- **Pi-native prompt compilation** — 0.20.2 rewrote the prompt path to use Pi's template engine directly. Custom string munging in `soma-boot` removed.
- **`_mind.md` template** — the user-editable template for the assembled system prompt. Variables (`{{soul}}`, `{{ecosystem}}`, etc.) interpolate from body files.
- **`_mind.md` orphan fix** — in 0.20.1.1 / 0.20.2.1, a Pi-native shortcut was accidentally bypassing `_mind.md`. Fixed; template compilation is now the single path.
- **`compileFrontalCortex` prepend** — behavioral rules + muscle memory auto-prepended onto `system-core.md` before the template renders. Don't re-render via `{{protocol_summaries}}` in templates (causes duplication — see SX-542).

### CLI restructure
- **Three-layer CLI** — `thin-cli.js` (lightweight entrypoint, ~200 LOC) → `personality.js` (welcome, init, model, first-run) → Pi `core/cli.js` (the actual agent). Published from `npm/` only.
- **`repos/cli/` retired** — merged into `repos/agent/npm/`. Archived at `repos/_archive/cli-repo-2026-04-18/`.
- **Shipping-integrity Layer 4** — `thin-cli.js:652` compares declared Pi (`package.json`) vs installed Pi (`node_modules/`) and warns on drift. Added after the `opus-4-7 missing` bug (s01-054a4c).
- **`soma update`** — unified upgrade path; replaces manual `npm install -g` dances.
- **Bundled script routing** — `soma <name>` walks bundled → project → global to find `soma-<name>.sh` / `.js`.

### Pro-script scaffolding
- **`scripts/_pro/`** — source-hidden scripts (`soma-refactor.sh`, `soma-seam.sh`, `soma-browser.sh`, etc.). Compiled to obfuscated `.js` at release time; sources never shipped. No runtime feature-gating — every user who installs meetsoma gets them working.

### Identity / body
- **`body/children/`** — new directory with role templates. Seeded by `soma init` / `soma doctor` if missing.
- **`_memory.md`** — preload template. Next-self briefings use this format.
- **`body/DNA.md`** — variable registry + template conventions.


**Actions:**
- No user-side action required — `addIfMissing` in `extensions/soma-boot.ts:1167-1178` adds new settings keys on next boot.
- Doctor tier-1 soft-check scaffolds missing files via `writeIfMissing` (never clobbers).
- See § Migration Steps below for optional opt-in features.

## Migration Steps

**None required.** Additive features; the chain is handled by:
- Boot-time `addIfMissing` in `extensions/soma-boot.ts:1167-1178` for settings
- Doctor seeding in `core/init.ts` for missing body files (`body/children/*.md`, `_tools.md`)

### Optional (activate delegation)

1. `soma update`
2. Dispatch: from within a Soma session, use the `delegate` tool with any role in `body/children/`.
3. Read role templates: `body/children/_child-template.md` explains the frontmatter.

### Optional (edit the system prompt template)

1. Edit `.soma/body/_mind.md` in your project.
2. `/rebuild` in session (or restart) to recompile.

## Compatibility

- Old `settings.json` — forward-compatible. New keys auto-added at boot.
- Old body files without `_tools.md` — tool registry uses defaults (all tools enabled, no overrides).
- Old body files without `body/children/` — `delegate` falls back to agent-bundled defaults under `~/.soma/agent/body/children/`.
- Custom extensions using `pi.registerTool` directly — still work. `somaRegisterTool` is additive.

## Known Issues (at 0.20.0 → carried into 0.20.3)

- `compileFrontalCortex` + `{{protocol_summaries}}` double-render (SX-542)
- Template interpolator doesn't skip YAML/HTML comments (SX-543)
- `/reload` cache invalidation ($1/invocation) — fixed in 0.20.3 via disk-backed prompt cache (SX-495)
