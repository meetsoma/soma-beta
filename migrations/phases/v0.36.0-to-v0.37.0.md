## Summary

The system prompt was split along a **scaffold-vs-behavior** line. `prompts/system-core.md`
(the always-loaded framework harness, the same for every install) is now **pure scaffold** — it holds
session mechanics, the memory tree, tool-discovery, docs, commands, rhythm, and *pointers* to the
behavioral layer. **All behavioral rules now live in `core_rules.md`** — a body file you own and tune,
ours-vs-theirs like `soul.md`/`voice.md`.

Four behaviors that used to live in system-core moved into the `core_rules` **seed**:
**Match the Codebase · Name the Approach · Mark Reversals, Don't Delete · Guard Secrets, Refuse Harm.**

## What Changed

- **system-core.md → pure scaffold.** Removed the behavioral bullets (match-codebase, run-before-theorize
  detail, verify-before-claim, crystallize-script, corrections, mark-reversals, name-approach, the
  run-wins/guard-secrets floor). They are replaced by one-line pointers to `core_rules`.
- **seed `core_rules.md` gained 4 sections** (Match the Codebase, Name the Approach, Mark Reversals,
  Guard Secrets) so the behavioral layer is complete on its own.
- Stale-doc fixes: warm muscles **and** protocols load `## TL;DR` (not the frontmatter `description`,
  which is only the fallback). The protocol template now uses `## TL;DR` (was `## Summary`).

## Actions — Migration Steps

> **Why `fix-mode: agent`, not a mechanical auto-update:** `core_rules.md` is **user-owned**. The
> boot-time template auto-updater deliberately skips non-`_` body files — it never clobbers `soul`,
> `voice`, or `core_rules`. So these behaviors can't be force-written into your file; they need a
> **semantic merge** that preserves your customizations.

> **Whatever version you're upgrading FROM, don't assume — diff.** You may be coming from well before
> v0.36.0, so your old system-core may never have carried these exact rules. The only thing that matters
> is the *end state*: after v0.37.0, ALL behavior lives in `core_rules` and system-core is pure scaffold.
> So migrate by **comparing the new files against yours**, not by assuming what you had.

**New installs / pristine `core_rules`:** nothing to do — the seed already carries the full behavior set.

**Customized `core_rules` (you've edited yours) — visualize the delta, then merge:**

1. **Diff the new `system-core.md`** (read it in the agent install: `prompts/system-core.md`). Confirm
   it's now scaffold + pointers, each ending *“…lives in `core_rules`.”* Those pointers name every
   behavior that should now live in your `core_rules` — the checklist for step 3.
2. **Diff the new `core_rules` seed against yours** — this *shows you the exact difference*:
   `diff <agent-install>/templates/default/core_rules.md <soma>/body/core_rules.md`
   The seed-only (left) sections are what you're missing — most importantly the four that moved out of
   system-core: **Match the Codebase · Name the Approach · Mark Reversals, Don't Delete · Guard Secrets.**
3. **Merge** the missing sections into your `core_rules.md`, adapting the wording to your voice. Leave
   every existing custom rule of yours untouched.

(An agent running this migration does steps 1–3 directly: read the new system-core (the pointer
checklist), diff the new seed against the user's `core_rules.md`, merge the missing behavior sections,
preserve all customizations, and report what it added.)

## Compatibility

- **Custom `core_rules.md`** — fully preserved (never auto-overwritten). The only action is the optional
  merge above.
- **Custom extensions / settings** — unaffected.
- **`_*.md` framework templates** (`_mind`, `_memory`) — auto-updated when pristine, kept when customized,
  as always.

## Verify

1. `prompts/system-core.md` (agent install) has **no** behavioral bullets — only scaffold + pointers
   ending in *"…lives in `core_rules`."*
2. Your `core_rules.md` contains (or has equivalents for) the four behaviors above.
3. `soma doctor` reports the workspace at v0.37.0.
