# v0.20.3 → v0.21.0 — Cache Economics + Discoverability + Self-Knowledge

> Written s01-f6e928 (2026-04-22). Covers the v0.21.0 release arc: cache
> stickiness (shipped 0.20.3), cache economics (shipped v0.21.0 via SX-547/
> SX-548/SX-549/SX-551), background delegation (SX-553 Phase A), lazy-load
> enforcement (SX-548), discoverability (SX-558), and self-knowledge /
> crystallize (SX-559 Phase A).
>
> Single public hop: v0.12.4 users jumping straight to v0.21.0 should read
> this map plus v0.12.0-to-v0.20.0 for the full story. Neither requires user
> action — `addIfMissing` handles settings migration at boot; the only
> optional user action is accepting an agent-driven check on
> `body/_memory.md` (see frontmatter `agent-prompt`).

## Summary

The v0.21.0 arc is three woven threads:

1. **Cache economics** — stop invalidating the cached prefix on `/reload`
   (SX-495, shipped 0.20.3), enforce lazy-load (SX-548), relocate Agent
   Notes out of `body/ecosystem.md` to `memory/notes/soma-log.md` (SX-547),
   auto-inject the last 3-5 ancestry entries + releases INDEX into the
   preload user-message so next-self sees continuity without paying the
   cached-prefix cost (SX-549, SX-551).
2. **Discoverability** — `soma tool <name>` CLI + `capabilities` Pi tool
   (SX-558) so agents (and users) can introspect the tool registry without
   re-reading the system prompt.
3. **Self-knowledge / crystallize** — `soma new muscle <name>` + `soma new
   protocol <name>` + canonical templates (SX-559 Phase A) so agents notice-
   then-crystallize-a-pattern flow goes from ~6 steps to 1.

Plus: background delegation (SX-553 Phase A), cache.retention setting
(SX-544), endpoint resolver (SX-513), children-dashboard registry fix
(SX-557, workspace-scope only), build-system hygiene, CLI ergonomics.

All `.soma/` changes are additive and self-healing at boot via
`addIfMissing`. No breaking changes. The one place a user-customized file
matters is `body/_memory.md`, where the bundled template gained a Recent
Ancestry reference in v0.21.0 — doctor offers to check the user's override
but never clobbers it.

## What Changed

### Agent Notes relocation (SX-547)
- **Old location:** `body/ecosystem.md § Agent Notes` (~166 lines, grew on every session-end append, invalidated the cached prefix every boot).
- **New location:** `memory/notes/soma-log.md`. Rolling session-observation log, entries top-to-bottom (newest first), per-agent subsections (Soma, Somaverse, Somadian).
- **`body/ecosystem.md`** now contains a pointer under `## Agent Notes` directing readers to the new location + `READ WHEN` / `WRITE WHEN` criteria. The file is `lazy: true` so even its shrunken body doesn't hit the cached prefix; only the `description` frontmatter renders.
- **Preload auto-inject** (SX-549): last 3 entries from `memory/notes/soma-log.md` are injected as `## Recent Ancestry` in the user-message on fresh boot. Configured via `settings.preload.recentNotesCount` (default 3; 0 disables).

### New settings keys (auto-migrated)
- `settings.preload.recentNotesCount` — default 3
- Applied at boot via `addIfMissing` + a sub-migration for existing `preload` blocks that predate the field.

### SX-551 deferred
- An earlier draft injected the top of `releases/INDEX.md` into every preload. Pulled back s01-f6e928 — that path is a meetsoma-specific plan-folder convention, not a generic user concept. The right long-term shape is a compiled `_memory.md` template that lets users declare their own injection variables. Design in `.soma/releases/v0.20.x/plans/preload-template-compiled/README.md` (SX-560). Ship target: v0.22.0 or later.

### Backfill-chain note (important for upgrades)
- Pre-0.21.0 users never received the backfilled migration chain (`v0.11.0-to-v0.12.0`, `v0.12.0-to-v0.20.0`, `v0.20.3-to-v0.21.0` — all written s01-f6e928). Doctor treats every upgrade to 0.21.0 as potentially missing the full chain's structural pieces.
- Tier-1 soft check in `soma-boot.ts:session_start` scaffolds missing pieces using `writeIfMissing` semantics (never clobbers):
  - `memory/notes/soma-log.md` (SX-547 destination)
  - Settings keys via `addIfMissing`
- Tier-2 (agent-driven, see `agent-prompt` above) reviews `body/_memory.md` and offers to add a Recent Ancestry reference if customized. Leaves untouched if user declines.

### Cache stickiness (SX-495, shipped 0.20.3)
- **Disk-backed prompt cache** — `.soma/state/.session-prompt-cache.json` holds the compiled system prompt. `/reload` rehydrates from disk instead of recompiling → no Anthropic cache miss.
- **`/rebuild` command** — explicit opt-in to recompile. Use this after editing `body/*.md` mid-session if the change should apply immediately. Otherwise edits land on next launch.
- **Severity-aware statusline indicator (line 3)** — replaces the intrusive "Changes detected" toast. Labels: `🔄 /reload` (extensions + core TS), `📝 /rebuild?` (body edits, optional), `⚠ relaunch` (dist/* or core JS).

### Background delegation (SX-553 Phase A, in `[Unreleased]`)
- **`delegate(background: true)`** — spawns a child soma in a new cmux pane and returns a handle (`{childId, pane, surface, status}`) immediately, instead of blocking. Synchronous mode still works (omit the flag).
- **`children` tool** — `op: "list"` reads `~/.soma/state/children.json` and formats a status table (id, role, model, status, cost, runtime, pane).
- **Atomic-rename IO helpers** — `appendChild`, `updateChild`, `readChildrenJson`, `writeChildrenJson`. Safe concurrent writes from multiple somas.

### Lazy-load enforcement (SX-548, in `[Unreleased]`)
- **`body/*.md` with `lazy: true`** — only the `description` frontmatter field interpolates into the cached prompt. Full body is accessed on-demand via the `read` tool. **Opt-in.** Missing `lazy: true` → same behavior as before.
- **`body/core_rules.md` (project-level)** — appended after the shipped `system-core.md` rather than replacing it. Shipped rules cannot be accidentally dropped by a project override. Missing file → identical behavior to before.

### Cache economics opt-in (SX-544 Phase A, in `[Unreleased]`)
- **`cache.retention` setting** — new key. Values: `"long"` (1h Anthropic TTL, 2× cacheWrite cost, break-even at one re-hit within 60m), `"short"` (default 5m), `"none"` (disabled — unusual), `null` (inherit shell env / Pi default).
- **Boot-time migration** — `addIfMissing("cache", { retention: null })` at `extensions/soma-boot.ts:1178`. Existing users get the key on next boot with no behavior change.

### Endpoint resolver (SX-513, in `[Unreleased]`)
- **Single source of truth for URLs** — `_shared/env.ts` (Node) + `soma-env.sh` (bash). Parity-tested. Tier 2 extensions (`bridge-connect`, `workspace-tools`) + dev scripts (`launch-browser`) route through it.
- **`environment.mode` setting** — `local` / `cloud` / `pro` / `enterprise` / `auto`. User default `cloud`; dev default `auto` (probes `localhost:18800`, falls back to cloud). Read only by dev extensions; prod extensions ignore it.
- **`bridge-connect-dev.ts`** — dev-only sibling extension with runtime `env_status` tool. Installed via `soma-dev symlink extensions --dev`.

### CLI ergonomics (in `[Unreleased]`)
- **`soma update --yes` / `-y`** — skips the Y/N prompt. Scriptable one-liner: `npm install -g meetsoma@latest && soma update --yes`.
- **`SOMA_PROJECT_DIR`** — env var exported to any `soma <cmd>` script; project context available without re-discovery.

### Script fixes (in `[Unreleased]`, all backward-compatible)
- `soma refactor` / `soma seam` — resolve_soma_dir works from any cwd; SIGPIPE handled.
- `soma plans` — bash 3.2 compat restored.
- `soma-install.sh stable` — emits an honest error instead of silent failure.


**Actions:**

## Migration Steps

**None required.** Every change is additive or boot-time self-healing.

### Optional (opt into 1h cache retention)

1. Edit `.soma/settings.json`:
   ```json
   { "cache": { "retention": "long" } }
   ```
2. Restart session. Confirmation: `cacheWrite` counter in statusline roughly doubles on first turn, then `cacheRead` stays warm across 1h idle gaps.
3. **Break-even:** one cache hit within 60 minutes. Long-running dev sessions hit this trivially.

### Optional (opt into lazy-loaded body files)

Edit a volatile body file (e.g. `.soma/body/ecosystem.md`) and add frontmatter:
```yaml
---
lazy: true
description: One-line summary that will replace the full file in the cached prompt.
---
```
The agent reads the full file via the `read` tool when it needs more. Reduces cached prefix size by the body-file length (often 5-10K tokens).

### Optional (opt into project-level core rules)

Create `.soma/body/core_rules.md`:
```md
## Project Rules

- Rule 1...
- Rule 2...
```
Appended after the shipped rules (not replacing them).

### Optional (background delegation)

Use `delegate(task: "...", role: "verifier", background: true)` from within a Soma session. Query status with the `children` tool.

## Compatibility

- Old `settings.json` — `cache`, `environment`, and new nested keys auto-added at boot. No breaking changes.
- Old extensions using `pi.registerTool` — still work.
- Old delegation call sites (no `background` flag) — synchronous behavior preserved.
- Projects without `.soma/body/core_rules.md` — identical to pre-0.20.4.
- Projects without `lazy: true` frontmatter — identical to pre-0.20.4.
