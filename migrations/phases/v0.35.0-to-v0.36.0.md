# Migration v0.35.0 → v0.36.0

Two user-data-relevant changes ship in this release, both auto-migrated at first v0.36.0 boot
(no `doctor --fix` interaction required). Plus the SX-804 tool-loading-hygiene arc (P1–P5) and the
SX-799/800/801 + SX-803 fixes, which are runtime-only (no user-data migration).

## What changed (user-data relevant)

### 1. Tool config: `body/_tools.md` → `settings.json` `tools.disabled` (SX-804 P3)

`_tools.md` is retired. Tool enable/disable now lives in `settings.json` under `tools: { disabled: [...] }`,
authoritative for ALL tools (filtered via `pi.setActiveTools` at session start — the old `_tools.md` gate
only caught `somaRegisterTool` tools, not raw `pi.registerTool`/meta-tools).

**Migration mechanism:** `applyOnceAlways("tools-to-settings-v0.35.x", …)` in `soma-boot.ts`:
- adds `tools: { disabled: [] }` to `settings.json` if absent;
- ports any `## Disabled` entries from an existing `body/_tools.md` into `settings.tools.disabled`;
- archives `_tools.md` → `body/_archive/_tools.md` (with a RETIRED banner) and removes the original.

A customized `_tools.md` is preserved verbatim in `_archive/`; nothing is lost. Idempotent (a missing
`_tools.md` is a no-op once `settings.tools` exists). **Template:** `templates/default/_tools.md` removed.

### 2. Muscle + automation heat: `.md` frontmatter → `state.json` (SX-804 P4)

Muscle/automation heat used to be written into each `.md` file's frontmatter every session (constant
`.soma` churn). It now lives in `state.json` (`muscles{}` / `automations{}` dicts, same shape as
`protocols{}`), mutated in-memory and persisted at exhale. Frontmatter `heat:` becomes a **static seed**
(read once to seed the first state entry, then never written).

**Migration mechanism:** automatic seeding at boot — `seedAndOverlayMuscleHeat` /
`seedAndOverlayAutomationHeat` (`core/{muscles,automations}.ts`) create a `state.json` entry per known
muscle/automation, seeded from its frontmatter `heat:`, and prune entries for ones that no longer exist.

**s01-499fab follow-up (the deterministic-persist fix):** seeding originally persisted only at exhale —
an instance that booted and exited without a clean exhale never got the `muscles`/`automations` dicts
written to `state.json` (it re-seeded from frontmatter each boot). The seed functions now report whether
they changed the state structure, and `soma-boot.ts` persists `state.json` once on that first boot. So
after the v0.36.0 update, `state.json` deterministically gains the `muscles`/`automations` dicts —
verified on a real instance. Per-session heat *bumps* still persist only at exhale
(that's the churn-avoidance); the save fires only on structural change (rare: first boot, or a muscle
added/removed).

**Stale frontmatter `heat:` is left in place by design** — it's the seed for fresh instances and is never
rewritten. No cleanup migration; the values are inert after the first seed.

### 3. Preload format: `_memory.md` template refined (SX-804 arc, dogfooded)

The `_memory.md` preload template gained three generic sections proven in dogfood — **`⛔ Ground Truth`**
(the anti-wander slot: settled facts the next self would otherwise re-derive), **`Open Loops`** (numbered
forward blockers, "start here if cold"), and **`Decisions Made`** (closed architectural choices) — plus a
consolidation of `Gaps`/`Unfinished`/`Prior Preloads` into an *Optional sections* note and a merge of
`Warnings`+`Traps` into one `Traps` section. `soma_template_version` 0.35.0 → 0.36.0; archived:
`templates/default/v0.35.0/_memory.md`.

**Migration mechanism:** `applyOnceAlways("template-auto-update-v0.36.0", …)` in `soma-boot.ts` — the same
archive-byte-match-pristine-only logic as the v0.35.0 sentinel. A project's `_memory.md` is overwritten with
the new bundled version ONLY if it byte-matches a known archive (`templates/default/v*/`) = provably
pristine-old; a customized `_memory.md` matches no archive → KEPT verbatim. New installs get the new format
as their seed.

## Actions (migration mechanism)

All three are `applyOnceAlways(...)` boot sentinels in `extensions/soma-boot.ts` (fire once on the first
v0.36.0 boot) + automatic seeding, gated on `status.needsMigration`. None require `doctor --fix` interaction.

- **`tools-to-settings-v0.35.x`** — backfills `settings.tools.disabled`, ports any `_tools.md ## Disabled`
  entries, archives `_tools.md` → `body/_archive/`. Idempotent.
- **heat seeding** — `seedAndOverlayMuscleHeat` / `seedAndOverlayAutomationHeat` (`core/{muscles,automations}.ts`)
  seed `state.json` `muscles{}`/`automations{}` from frontmatter at boot; `soma-boot.ts` persists `state.json`
  once when the structure first appears (deterministic, not exhale-dependent). Stale frontmatter `heat:` left as seed.
- **`template-auto-update-v0.36.0`** — overwrites a project's `_memory.md` with the new bundled version ONLY
  if it byte-matches a known archive (`templates/default/v*/`) = pristine; a customized copy is KEPT.

Verified end-to-end on a real instance: all three fired correctly (tools archived +
settings.tools added, 24 muscles seeded into state.json, `_memory.md` 0.35.0 → 0.36.0).

## What did NOT need a migration

- SX-804 P1/P2/P5 (extension loader hygiene, somaverse Tier-2 consolidation) — runtime/dev-tooling only.
- SX-799/800/801, SX-803, system-core prompt — runtime behavior; no user-data shape change.
