# v0.21.1 → v0.21.2 — Bundled Template Backport + Version Stamp Fixes

> Written s01-ce1734 (2026-04-23). Covers the v0.21.2 fixes that surfaced
> during arzadon dogfood + SX-598 pre-flight audit.

## Summary

Three content improvements to bundled body templates — backporting v0.21
cache-economics + structural patterns that shipped to meetsoma but never
reached the bundled defaults users scaffold from. Plus two version-stamp
fixes that make `soma init` correctly stamp current agent version.

**Nothing breaking.** Existing projects keep working. Pristine-unmodified
bundled copies get an optional refresh offer; user-edited files are left
alone. Doctor already auto-bumps stale `settings.json.version`.

## What changed (v0.21.2 release)

### Bundled template content (SX-600)

- **`templates/default/_mind.md`** — rewrite to match v0.21 cache-economics
  pattern. Sticky identity slots (soul / voice / body / ecosystem) early;
  `{{core_rules}}` late (1% invalidation cost); volatile slots
  (`context_awareness` / `date_time_cwd` / `inbox_summary`) last. Added
  `{{#ecosystem}}...{{/ecosystem}}` conditional so ecosystem section hides
  cleanly when file absent. **Removed duplicate** `{{protocol_summaries}}` /
  `{{muscle_digests}}` / `{{scripts_table}}` interpolation — those are
  already prepended by `compileFrontalCortex`, so previous bundled users
  got duplicated behavioral content in their system prompt.
- **`templates/default/_memory.md`** — added `Prior Preloads` + `Kanban
  Snapshot` sections (marked optional; "skip if not applicable"). Generalized
  from meetsoma's evolved preload format.
- **`templates/default/DNA.md`** — added conditional blocks syntax, identity
  resolution chain explanation, "variables only work inside `_` templates"
  footgun warning, Known Quirks mini-section (mtime-based preload pickup,
  CWD rename breaks bash, template chain priority, resume skips `_boot.md`).
  Updated slash-command references (`/body vars` → `soma body vars`).

### Version stamp correctness (SX-619 + SX-620)

- **`scripts/build-dist.mjs`** — copies `package.json` into `dist/` so
  `init.ts` `AGENT_VERSION` resolves via `import.meta.url` walk-up. Prior
  builds omitted this; `soma init` silently fell through to the hardcoded
  `"0.10.0"` catch branch.
- **`core/init.ts`** — always override `settings.version` with current
  `AGENT_VERSION` even when a user-provided template at
  `~/.soma/templates/init/settings.json` or `<workspace>/.soma/templates/init/`
  declares its own `version` field. User templates are useful for paths /
  persona / inherit flags, but the version field must be runtime-stamped or
  it goes stale on every agent release.

## User-visible impact

### For fresh `soma init` after upgrade

Nothing to do. New projects scaffold with:
- Cache-optimized `_mind.md` (no duplicate content, volatile slots last)
- Version stamped correctly (matches `soma --version`)
- Improved `DNA.md` + `_memory.md` formats

### For existing projects with pristine bundled body/ files

Doctor runs this migration map. For each of `_mind.md`, `_memory.md`, `DNA.md`:
- If pristine bundled (unmodified since scaffold), offers interactive refresh
- If user-edited, leaves alone; suggests manual diff if desired

### For existing projects with stale settings.json version

Doctor's existing bump logic handles this (seen in the
`⚠ Project .soma/ is at vX.Y, agent is at vZ.W` + `Version bumped` flow).
No new code needed — the SX-619/SX-620 fixes affect NEW scaffolds; existing
drift falls through the already-working doctor path.

### For users with stale user-templates (e.g. hardcoded `version: "0.6.4"`)

SX-620 means the version field in user-templates is ignored on scaffold —
current agent version always stamps. Other fields (paths, persona, inherit)
still honored. If the user-template is meaningfully stale (not just version),
consider deleting it: it was useful for a prior agent version.

## Actions

### Automatic (via existing doctor paths)

- **Settings version bump** — doctor's `projectDoctor` existing logic detects
  `projectV < agentV` and bumps `settings.json.version`. Works today without
  any migration-map wiring.
- **Fresh scaffolds** — SX-619 (`dist/package.json` copy) + SX-620 (runtime
  override of user-template `version` field) land in the release build.
  Every `soma init` after this version ships correctly-stamped settings.

### Narrative (agent-prompt driven, not yet wired to doctor execution)

The `agent-prompt` frontmatter block above documents the intended
pristine-refresh behavior for body templates. **As of v0.21.2, that prompt
is narrative only** — the migration-replay mechanism in `thin-cli.js`
only processes `## Doctor Actions` JSON blocks (settings-defaults,
settings-subkeys, scaffold-files via writeIfMissing). Agent-prompt fields
are human/agent readable guidance, not yet invoked programmatically.

SX-621 (shipped this release) took a simpler path: hash-based auto-
refresh in doctor directly, keyed off `migrations/template-hashes.json`.
No agent-interactive round-trip; pure hash diff + narrow decision tree
(pristine-older → auto-replace; customized → warn with diff path;
current → skip).

### Manual path (fallback when doctor is bypassed)

The SX-621 auto-refresh handles the common case. For users who want to
force-adopt bundled updates on a customized file (declining the auto-
refresh is non-destructive; rm + soma init --force is the escape hatch):

```bash
# From project root:
rm .soma/body/_mind.md .soma/body/_memory.md .soma/body/DNA.md
soma init --force
```

`soma init --force` uses `writeIfMissing` semantics — fills the deleted
files from current bundled templates without touching any of the user's
other body files (soul.md, voice.md, etc.).

## Future — SX-599 replaces this pattern

Once SX-599 (global body layer) ships in v0.22.x, bundled-template refresh
becomes automatic via `~/.soma/body/` being a normal chain-walk layer with
`settings.scaffold.templateHashes` for authoritative drift detection. This
migration file is a transitional mechanism — every subsequent bundled
template improvement no longer needs its own migration file once SX-599 is in.
