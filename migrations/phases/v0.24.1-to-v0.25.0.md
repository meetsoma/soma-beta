# Migration: v0.24.1 → v0.25.0

## Summary

**Stale `body/_mind.md` templates get auto-cleaned** of redundant
`{{protocol_summaries}}` / `{{muscle_digests}}` / `{{scripts_table}}`
variable interpolations. The originals are backed up; sentinel ensures
the migration runs exactly once per soma.

This is the user-side completion of `fcd32bd` (SX-600, v0.21.0), which
fixed the shipped template but couldn't reach into existing user
workspaces because `addIfMissing` is additive-only.

## Why

`compileFrontalCortex()` in `core/prompt.ts` prepends the static behavioral
block (protocol summaries + muscle digests) BEFORE the user's `_mind.md`
template renders. Interpolating these variables INSIDE `_mind.md` causes
**double injection** — the same content lands in the prompt twice.

Verified empirically (s01-86b0fd workspace audit):
- `~/.soma/` (v=0.9.0) — affected
- `lab/.soma`, `lab/ai-video-workflows/.soma`, `lab/Soma3D/.soma` (all v=0.9.0) — affected
- `meetsoma/repos/agent/.soma` (v=0.10.0) — affected
- `meetsoma/somaverse/.soma` (v=0.20.3) — affected (one version pre-fix)

All exhibited the same pattern: bare-line interpolation
(`^\s*{{<var>}}\s*$`). No project had inline-prose usage.

Cost (per-session, per-affected-soma):
- ~3,719 tokens for `{{muscle_digests}}` re-injection
- ~3,634 tokens for `{{protocol_summaries}}` re-injection
- ~1,069 tokens for `{{scripts_table}}` re-injection
- **~8,422 tokens total redundancy** (~27% of typical context load)
- Plus cache-bust impact: line-21 position invalidates ~80% of prefix on
  any muscle/protocol edit

## Behavior

### For new installs (fresh `soma init`)

No change. Shipped template (`templates/default/_mind.md`) has been clean
since `fcd32bd` (2026-04-23, v0.21.0). The migration sentinel is recorded
on first boot; the migration is a no-op for already-clean templates.

### For existing users with stale `_mind.md`

On next boot OR next `soma doctor` invocation:

1. Tier 1 detects bare-line `{{protocol_summaries|muscle_digests|scripts_table}}` interpolations in `<soma>/body/_mind.md`.
2. Backs up the original to `<soma>/body/_mind.md.bak-v0.25.0` (preserves existing backup if present — never overwrites).
3. Removes the offending bare lines.
4. Collapses any 3+ runs of blank lines back to 2 (preserves paragraph breaks).
5. Inserts a one-line breadcrumb HTML comment near the top of the file documenting the migration date + reason + backup location.
6. Appends `mind-prepend-cleanup-v0.25.0` to `settings.migrations[]`.
7. `soma doctor` prints a one-line notice; soma-boot logs to debug only (no boot-time noise).
8. Subsequent boots skip the migration (sentinel present).

### For users who manually re-add the variables

If the user later puts the variables back into their `_mind.md`, the
sentinel respects that choice — the migration does NOT run again. They've
opted in to double-injection knowingly.

### For users on already-clean templates

The migration runs once, finds no offending lines, records the sentinel,
exits in ~1ms. Idempotent.

## Actions

**soma will automatically apply this migration on next boot OR `soma doctor`.**
No manual steps required.

Migrations now run UNCONDITIONALLY (s01-86b0fd lifted them out of the
version-mismatch gate). Previously, migrations were gated behind
`if (status.needsMigration)`, missing users whose project version already
matched the agent. Now `applyOnce()` migrations run on every boot/doctor
invocation; the sentinel makes them idempotent (returns early after first
application, regardless of work performed).

1. At boot, `soma-boot.ts` reads `settings.migrations[]`.
2. If sentinel `mind-prepend-cleanup-v0.25.0` is absent, run the migration
   block (read `body/_mind.md`, scan for bare-line redundant vars, backup
   + clean if needed, OR no-op if already clean).
3. Append the sentinel either way — future invocations skip in O(1).
4. `soma doctor` mirror runs identical logic via `npm/thin-cli.js`.
5. Pre-flight prompt (s01-86b0fd, this version): if `updateAvailable` is
   cached in `~/.soma/config.json`, doctor surfaces the upgrade prompt
   before continuing. See `docs/troubleshooting.md` and the
   `2026-05-04-preflight-prompt-and-pi-cruft-cleanup.md` plan.

### `~/.soma/agent/` worktree concerns

None. The agent worktree is overlaid by `git pull` on ship — user state
lives in sibling dirs (`~/.soma/body/`, `~/.soma/settings.json`). The
migration touches user state only.

## Decision tree

```
soma-boot Tier 1 (and `soma doctor` mirror):

  has settings.migrations[] include "mind-prepend-cleanup-v0.25.0"?
    YES → SKIP entirely
    NO  → proceed

  read <soma>/body/_mind.md
    missing → record sentinel, return false (no fix needed)
    present → continue

  scan for bare-line `^\s*\{\{(protocol_summaries|muscle_digests|scripts_table)\}\}\s*$`
    0 hits → record sentinel, return false (already clean)
    ≥1 hits → continue

  backup file path: <soma>/body/_mind.md.bak-v0.25.0
    exists already → preserve, do NOT overwrite
    not present    → write current content + timestamp comment

  apply cleanup:
    1. remove bare-line var interpolation lines
    2. collapse 3+ consecutive blank lines → 2
    3. insert breadcrumb comment near top (after frontmatter if present)

  write modified _mind.md
  append "mind-prepend-cleanup-v0.25.0" to settings.migrations[]
  return true (a fix was applied)
```

## Migration Steps

**Mechanical — handled by Tier 1 / `soma doctor`. Nothing for the user
to do.** If you have a project with stale interpolations in
`body/_mind.md`, they will be removed on next boot or doctor run.

If you actively want the variables back (you've disabled the compiler
prepend, or you're doing custom prompt assembly), the migration sentinel
respects your edits — re-adding them won't be undone.

If you want to inspect what was changed:

```bash
diff <soma>/body/_mind.md.bak-v0.25.0 <soma>/body/_mind.md
```

The backup is byte-equivalent to your original pre-migration state.

| `_mind.md` with bare-line `{{protocol_summaries}}` | line removed; backup at `.bak-v0.25.0` | Compiler prepend fills the slot |
| `_mind.md` with bare-line `{{muscle_digests}}` | line removed; backup at `.bak-v0.25.0` | Compiler prepend fills the slot |
| `_mind.md` with bare-line `{{scripts_table}}` | line removed; backup at `.bak-v0.25.0` | Compiler prepend fills the slot |
| Custom user sections (headings, prose, other vars) | preserved exactly | Bare-line removal only |
| Variables inside HTML comments | preserved | Comments don't match the bare-line regex |
| Inline prose mentioning `{{var}}` (rare) | preserved | Bare-line regex requires whole-line match |

## Verify

After your next `soma` boot or `soma doctor`:

1. Check `<soma>/settings.json` — `migrations[]` should include
   `mind-prepend-cleanup-v0.25.0`.
2. Check `<soma>/body/_mind.md` — should not contain bare-line
   `{{protocol_summaries}}` / `{{muscle_digests}}` / `{{scripts_table}}`.
3. If a backup was created, `<soma>/body/_mind.md.bak-v0.25.0` exists
   alongside the cleaned template.
4. `soma body slots` no longer shows the redundant variables at line 21
   (or wherever they previously sat in your template).
